## 1. Project Reconnaissance

**Purpose:** macOS menu-bar agent that reads the frontmost app's menus via the Accessibility API and displays keyboard shortcuts in a floating popup.

**Entry points:**
- `KeyMinderApp.swift` (`@main`) — delegates immediately to `AppDelegate`
- `AppDelegate.swift` — status-bar icon (left/right click), global hotkey via Carbon, double-tap modifier via CGEventTap

**Privilege level:** Standard user. No admin rights required. Requires the **TCC Accessibility permission** (`AXIsProcessTrusted()`) to read other apps' menus. This is a per-user grant stored in `/Library/Application Support/com.apple.TCC/TCC.db` and is identity-anchored to the signing certificate.

**Entitlements:** None. No `.entitlements` file exists; no `com.apple.security.*` keys are set. This is correct for non-sandboxed Developer ID distribution — App Sandbox cannot coexist with Accessibility API cross-process reads.

**Info.plist security-relevant keys** (set via `INFOPLIST_KEY_*` in `project.pbxproj`):
- `LSUIElement = YES` — agent-only app (no Dock icon, no main menu bar). Correct.
- No `NSAppTransportSecurity` keys — no network stack exists.
- No `CFBundleURLSchemes` — no URL scheme handler.
- No `NSAppleEventsUsageDescription` / `NSAppleScriptEnabled` — no AppleScript/AppleEvents surface.

**Code signing:**
- Hardened Runtime: `ENABLE_HARDENED_RUNTIME = YES` on **both** Debug and Release. ✓
- Debug config: `CODE_SIGN_IDENTITY = "Apple Development"` — correct for local development.
- **Release config: `CODE_SIGN_IDENTITY = "Apple Development"`** — this should be `"Developer ID Application"` for notarization. See **F-05** below.
- `CODE_SIGN_STYLE = Automatic` on all configs; no `OTHER_CODE_SIGN_FLAGS`.
- No runtime exceptions in Hardened Runtime (e.g., JIT, unsigned executable memory, DYLD variables). ✓

## 4. Inter-Process Communication & Attack Surface

| Mechanism | Present | Notes |
|---|---|---|
| XPC services | No | — |
| NSXPCConnection | No | — |
| Distributed Objects | No | — |
| Mach ports (explicit) | No | — |
| CFMachPort (CGEventTap) | Yes | `DoubleTapTrigger` — listenOnly, read-only |
| Unix sockets | No | — |
| URL schemes | No | — |
| AppleEvents | No | — |
| Privileged helpers (SMJobBless) | No | — |
| Carbon global hotkey | Yes | `RegisterEventHotKey` — activates popup |
| NSEvent global monitor | Yes | `addGlobalMonitorForEvents(.keyDown)` — Esc only |
| AX cross-process read | Yes | `AXUIElementCopyAttributeValue` — read-only menus |
| AX cross-process action | Yes | `AXUIElementPerformAction(kAXPressAction)` — user-initiated |

**CGEventTap (`DoubleTapTrigger`):**
- Tap type: `.cgSessionEventTap`, `.listenOnly` — cannot suppress or synthesize events. ✓
- Events of interest: `flagsChanged` + `keyDown` only.
- The `keyDown` handler only resets the double-tap state machine; it does not inspect or log the key event content. ✓
- The `flagsChanged` handler masks to four modifier bits and ignores all other content. ✓
- `tapDisabledByTimeout` is handled by re-enabling the tap (correct macOS practice). ✓

**Global keyDown event monitor (`PopupController.installDismissalMonitors`):**
- Registered via `NSEvent.addGlobalMonitorForEvents(matching: [.keyDown])` while the popup is visible.
- This callback **receives all system-wide keyDown events** while the popup is open.
- The implementation correctly only acts on `keyCode == 53` (Esc).
- **No key content is logged, stored, or forwarded.** ✓
- The narrow time window (popup visible) limits exposure.
- The security contract is not documented in the code, creating a maintenance risk if the callback is extended.

**Finding ID: F-07 (Informational)** — a security comment to document the scope is proposed in `Security_Fixes_2026-06-01.md`, not yet applied.

**AX action (`ShortcutActivator`):**
- `AXUIElementPerformAction(element, kAXPressAction)` executes a menu item action in the target app.
- The `axElement` is a live reference obtained during the scrape of the frontmost app's menus.
- This action is only triggered by explicit user interaction (click or Return key on a selected row).
- If the target app exits between scrape and activation, the AX call fails gracefully (returns an error, logged at `.error`, no crash). ✓
- PID recycling risk is negligible: the frontmost-app change triggers popup dismissal before any other app can claim the PID.

## 5. Data Handling

**Shell / process execution:** None. No `Process`, `NSTask`, or shell invocation anywhere in the codebase. ✓

**File I/O:** None beyond AppKit/SwiftUI implicit (e.g., `NSImage` loading from asset catalog). No user-supplied file paths. ✓

**Path traversal:** Not applicable — no file-path construction. ✓

**Unsafe Swift (`Unmanaged`, raw pointers):**

Two instances of `Unmanaged.passUnretained` for bridging to C callbacks:

1. `HotkeyManager.swift:71` — passes `self` (a singleton) to Carbon's `InstallEventHandler` as `userData`.
2. `DoubleTapTrigger.swift:107` — passes `self` (a singleton) to `CGEvent.tapCreate` as `userInfo`.

Both are correct: `passUnretained` is appropriate because the C runtime does not take ownership, and the singleton lifetimes exceed the callbacks' lifetimes by design. The callbacks use `takeUnretainedValue()` — no retain/release imbalance. ✓

**Force cast (`as!`):**

`MenuScraper.swift:151`:
```swift
return (raw as! AXUIElement)
```

The surrounding guard checks `CFGetTypeID(raw) == AXUIElementGetTypeID()`, making the cast safe in normal operation. However, if the AX runtime returns an unexpected type (pathological AX state, future API change), this is a hard crash rather than a graceful `nil`. The return type is already `AXUIElement?`, so swapping to `as?` is behaviour-equivalent on the success path.

**Finding ID: F-01 (Low)** — proposed fix in `Security_Fixes_2026-06-01.md`, not yet applied.

**Recursive AX traversal without depth limit (NEW):**

`MenuScraper.collectShortcutsFlat` recurses into sub-submenus without a depth bound:

```swift
private static func collectShortcutsFlat(in menu: AXUIElement, includeAll: Bool = false) -> [Shortcut] {
    for item in children(menu) {
        ...
        if let submenu { result.append(contentsOf: collectShortcutsFlat(in: submenu, ...)) }
    }
}
```

A malicious or misbehaving application could construct an arbitrarily deep AX menu hierarchy. Each level of recursion makes a synchronous AX IPC call (bounded to 1.0 s by `AXUIElementSetMessagingTimeout`). The result is:

- **Stack exhaustion:** hundreds of levels of recursion on a `Task.detached` thread (default ~8 MB stack) could overflow the stack, crashing KeyMinder.
- **Prolonged blocking:** even without overflow, each level costs up to 1 s, so a 100-level deep tree blocks the scrape task for up to 100 s.

The scrape task is detached from the main thread, so the UI remains responsive (the popup would show stale or empty content), but KeyMinder would crash on stack overflow.

**Finding ID: F-NEW-1 (Low)** — add a depth parameter defaulting to 0 and bail at a reasonable limit (e.g., 10).

**UserDefaults tamper:**
A local attacker with filesystem access could modify `~/Library/Preferences/org.afaik.KeyMinder.plist` to change the hotkey configuration. The hotkey value is decoded via `JSONDecoder` into a typed struct (`GlobalHotkey`) with only `UInt32` and `String` fields — no code execution path from tampered values. The worst outcome is KeyMinder registers a different hotkey. This is an out-of-scope scenario (attacker already has local file access). Informational only.

## 7. Update & Distribution Mechanism

**No auto-updater.** No Sparkle, Squirrel, or custom update mechanism. Users update manually.

**Distribution path:** Developer ID + Hardened Runtime + notarization (documented in `CLAUDE.md`). Hardened Runtime is correctly enabled on the Release config. ✓

**Release signing identity mismatch (OPEN — F-05 from prior review):**
The Release build configuration (`3E3EC5A646DA464EBE9375A9`) has:
```
CODE_SIGN_IDENTITY = "Apple Development";
"CODE_SIGN_IDENTITY[sdk=macosx*]" = "Apple Development";
```
An archive built with `"Apple Development"` identity cannot be submitted for notarization. The Release config should use `"Developer ID Application"`. With `CODE_SIGN_STYLE = Automatic`, Xcode will fall back gracefully if the cert isn't in the local keychain, so this is a distribution-workflow footgun rather than an active security regression.

**Finding ID: F-05 (Informational)** — proposed fix in `Security_Fixes_2026-06-01.md`, not yet applied.

- No `SecKeychainFind*` (deprecated keychain APIs). ✓

## Top-5 Prioritised Remediation List

### 1. F-02 — Log privacy: third-party menu content (Medium)
**File/lines:** `MenuScraper.swift:90`, `MenuScraper.swift:130`, `ShortcutActivator.swift:18`

Change `privacy: .public` to `privacy: .private` for the `title` and `hint` string interpolations in the three affected log calls. This prevents other apps' menu structure from appearing unredacted in sysdiagnose archives. The change is one word per affected interpolation and has no runtime behaviour impact. Full change set and rationale in `Security_Fixes_2026-06-01.md` (Fix 1).

### 2. F-NEW-1 — Recursive AX traversal without depth limit (Low)
**File:** `MenuScraper.swift`, `collectShortcutsFlat(_:includeAll:)`

Add a `depth: Int = 0` parameter and return early at a limit (e.g., 10):

```swift
private static func collectShortcutsFlat(
    in menu: AXUIElement, includeAll: Bool = false, depth: Int = 0
) -> [Shortcut] {
    guard depth < 10 else { return [] }
    ...
    if let submenu {
        result.append(contentsOf: collectShortcutsFlat(in: submenu, includeAll: includeAll, depth: depth + 1))
    }
}
```

Ten levels of nesting is vastly more than any real macOS app uses, and it caps both stack depth and per-scrape latency.

### 3. F-01 — Force cast `as! AXUIElement` (Low)
**File:** `MenuScraper.swift:151`

Replace `return (raw as! AXUIElement)` with `return raw as? AXUIElement`. The surrounding `CFGetTypeID` guard makes the crash impossible in practice, but `as?` is the correct idiom for an optional-returning helper and eliminates the crash path entirely. Full change set in `Security_Fixes_2026-06-01.md` (Fix 2).

### 4. F-05 — Release config signing identity (Informational)
**File:** `KeyMinder.xcodeproj/project.pbxproj`, UUID `3E3EC5A646DA464EBE9375A9`

Change both `CODE_SIGN_IDENTITY` lines in the Release build configuration from `"Apple Development"` to `"Developer ID Application"`. This ensures archives are signed with the correct identity for notarization without any manual step at release time. Full change set in `Security_Fixes_2026-06-01.md` (Fix 3).

### 5. F-07 / F-04 — Security contract documentation and log annotation (Informational)
**Files:** `PopupController.swift:347–348`, `SettingsView.swift:77`

- Add a two-line security comment before the global `keyDown` event monitor registration to document that the callback receives all system-wide key events and must never log or store them (Fix 5 in `Security_Fixes_2026-06-01.md`).
- Add `privacy: .private` to the `error.localizedDescription` interpolation in the login-item error log to make the intent explicit (Fix 4).
