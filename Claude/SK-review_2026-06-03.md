## Issue 1 — All-entry mode loses content when no shortcuts exist

**Verdict: Valid bug. Fix it.**

SK's description is accurate. In `PopupController.measuredContent`, when
`includesItemsWithoutShortcuts` is true, `shortcutsOnly(app.sections)` may return
`[]` (e.g. an app where every menu item has no key equivalent). This flows into:

```swift
let distributedLayout = MenuLayout.distribute([], columns: 0)  // → []
let actual = max(1, distributedLayout.count)                   // → 1 (width OK)
let displayColumns = distributedLayout.map { … }               // → []  ← BUG
```

`actual` is clamped to 1 so the panel width is fine, but `displayColumns` is
empty because `.map` on an empty array is empty. `PopupFilterModel` gets no
columns, so the popup opens with the header but no content — even though
`app.sections` has real data.

The fix and the Issue 4 fix are intertwined; see the combined prompt below.

## Issue 2 — Hotkey saved before Carbon registers it

**Verdict: Valid bug. Fix it.**

`SettingsModel.handleKey` (line ~145):

```swift
hotkey = newHotkey                              // 1. UI updated
UserDefaults.standard.globalHotkey = newHotkey  // 2. Persisted
HotkeyManager.shared.register(newHotkey)        // 3. May fail silently
stopRecording()
```

`HotkeyManager.register` logs failures but returns `Void`. If Carbon rejects the
hotkey (e.g. ⌘Space is owned by Spotlight), the Settings badge shows the new
shortcut, it persists to `UserDefaults`, reloads on relaunch — and never fires.
The user has no signal that anything went wrong.

Note: registration failure is rare in practice (only system-owned chords), but
silent data corruption in settings is always worth fixing.

## Issue 3 — Rapid toggles accumulate concurrent AX scrapes

**Verdict: Directionally correct; severity slightly overstated. Fix it.**

SK is right that `Task.cancel()` on a `Task.detached` wrapping synchronous C AX
IPC has no effect mid-flight — the IPC runs to completion or timeout regardless.
On rapid toggling, each new `presentPopup()` starts a fresh detached task before
the previous one finishes, so you can accumulate as many concurrent AX traversals
as there are rapid toggles (bounded in practice by the 1 s `AXUIElementSetMessagingTimeout`).

SK's framing is slightly strong because the stale-result guard (`Task.isCancelled`
on the outer task) correctly prevents any concurrent result from reaching the UI.
The issue is wasted AX IPC load, not UI correctness. The CLAUDE.md comment
already documents the limitation, but it understates it: it says "at most one
scrape is in flight at a time once any previous scrapeTask is cancelled" — that's
not quite right, because the detached task is still running even after cancellation.

The fix doesn't need a full serial coordinator. The simplest correct approach:
move the detached scrape *inside* the cancellable outer task, drain any in-flight
scrape before starting a new one, and check cancellation immediately after the
drain. This ensures at most one AX traversal runs at any time.

## Issue 4 — `Dictionary(uniqueKeysWithValues:)` crash on duplicate section titles

**Verdict: Valid crash risk. Fix it.**

```swift
let fullByTitle = Dictionary(uniqueKeysWithValues: app.sections.map { ($0.title, $0) })
```

`Dictionary(uniqueKeysWithValues:)` traps on duplicate keys. `MenuSection` titles
come from the AX bridge reading arbitrary apps' `NSMenu` title strings — Apple
doesn't enforce uniqueness in `NSMenu`, and plugin architectures or localization
edge cases can produce two menus with the same title. Rare but realistic, and a
crash in production is always unacceptable.

SK suggests mapping by stable section index. I prefer a cleaner refactor: change
`shortcutsOnly` to return `(layout: MenuSection, full: MenuSection)` pairs so the
layout-to-full correspondence is structural rather than title-based. Then key the
lookup by `UUID` (the layout section's `.id`), which is stable and unique by
construction. This eliminates both the crash (Issue 4) and the empty-column case
(Issue 1) in one change.

3. Replace `Dictionary(uniqueKeysWithValues: app.sections.map { ($0.title, $0) })`
   with a dictionary keyed by the layout section's `.id` (UUID), derived from the

**Prompt:**

```
In KeyMinder/UI/Popup/PopupController.swift, fix two related bugs in
`measuredContent` and `shortcutsOnly`.

Bug A (Issue 1): when `app.includesItemsWithoutShortcuts` is true and

equivalent), `distributedLayout` is also empty, so `displayColumns` is `[]` and
`PopupFilterModel` has no columns to display — even though `app.sections` has
content. The popup renders as an empty panel.

Bug B (Issue 4): `Dictionary(uniqueKeysWithValues: app.sections.map {
($0.title, $0) })` crashes if two top-level sections share the same title.  AX
data comes from arbitrary third-party apps where title uniqueness is not enforced.

Fix both bugs together:

1. Change the private static `shortcutsOnly(_ sections: [MenuSection])` method to
   return `[(layout: MenuSection, full: MenuSection)]` instead of `[MenuSection]`.
   Each tuple holds the shortcuts-only copy (layout) alongside its originating
   full section (full). The implementation is the same compactMap, but instead of
   returning just the layout section, return `(layout: MenuSection(title:
   section.title, groups: groups), full: section)`.

2. In `measuredContent`, replace the block that computes `layoutSections`,
   `distributedLayout`, `fullByTitle`, and `displayColumns` with the following
   logic:

   a. Call the updated `shortcutsOnly` to get `sectionPairs: [(layout:
      MenuSection, full: MenuSection)]` when in all-entries mode. In normal mode,
      map `app.sections` to self-pairs: `app.sections.map { (layout: $0, full: $0) }`.

   b. If `app.includesItemsWithoutShortcuts` is true but `sectionPairs` is empty
      (no shortcuts exist at all), fall back to self-pairs of `app.sections` so
      the popup has content to display.

   c. Extract `layoutSections = sectionPairs.map(\.layout)` and use that for the
      existing `count` / `distributedLayout` / `actual` / `contentWidth` / `width`
      computation (unchanged logic).

   d. Build the full-section lookup as:
      `let fullByID = Dictionary(uniqueKeysWithValues:
          sectionPairs.map { ($0.layout.id, $0.full) })`
      This is keyed by UUID (guaranteed unique) rather than title string.

   e. Build `displayColumns` as:
      `distributedLayout.map { col in col.compactMap { fullByID[$0.id] } }`

No other callers of `shortcutsOnly` exist; no test files reference it. The
`MenuSection.id` property is a `let UUID` generated at init, so each layout copy
has a stable, distinct ID from the full section — the `fullByID` lookup will
always resolve correctly because `distributedLayout` contains the same
`MenuSection` values (by value equality including UUID) as `sectionPairs.map(\.layout)`.

Do not change any other method. Do not bump the version.

### Change C — AppDelegate: prevent concurrent AX traversals

Affects: `AppDelegate.swift` (`presentPopup`, `detachedScrapeTask`)

**What to change:**

Move the detached scrape inside the cancellable outer task. Before starting a new
AX traversal, await the in-flight one (if any) so it can drain. Check cancellation
immediately after the drain so a rapid second toggle discards the result rather
than kicking off another traversal. This guarantees at most one AX traversal runs
at any moment, without a full serial queue.

**Prompt:**

```
In KeyMinder/AppDelegate.swift, fix `presentPopup` so that concurrent AX
traversals cannot accumulate during rapid popup toggles.

The current bug: `Task.cancel()` on a `Task.detached` that wraps synchronous C AX
IPC (MenuScraper.scrape) has no effect mid-flight. Each rapid call to
`presentPopup` creates a new detached task while the previous one is still running,
so multiple AX traversals can run simultaneously. Results are discarded correctly
(via the `Task.isCancelled` guard on the outer task), but the wasted IPC load
grows with toggle frequency.

Refactor `presentPopup` as follows:

1. Remove the `private var detachedScrapeTask: Task<[MenuSection], Never>?`
   property entirely.

2. Rewrite `presentPopup` so that the detached scrape is created *inside* the
   outer `scrapeTask`, not before it. The new structure:

   ```swift
   private func presentPopup() {
       // ... guard AccessibilityPermission, guard frontmostApp (unchanged) ...
       // ... capture pid, appName, bundleID, icon, includeAll (unchanged) ...

       scrapeTask?.cancel()

       scrapeTask = Task {
           // If a previous scrape is still running (it can't be interrupted),
           // wait for it to finish before starting a new one.  The result is
           // discarded; we only wait so we don't pile up concurrent AX traversals.
           if let previous = self.detachedScrapeTask {
               _ = await previous.value
               self.detachedScrapeTask = nil
           }

           // If we were cancelled while waiting (another presentPopup fired),
           // stop here — the newer task will handle the scrape.
           guard !Task.isCancelled else { return }

           let work = Task.detached(priority: .userInitiated) {
               MenuScraper.scrape(pid: pid, includeItemsWithoutShortcuts: includeAll)
           }
           self.detachedScrapeTask = work

           let sections = await work.value
           self.detachedScrapeTask = nil

           guard !Task.isCancelled else { return }

           // ... build AppShortcuts and call popup.show (unchanged) ...
       }
   }
   ```

   Keep `private var detachedScrapeTask: Task<[MenuSection], Never>?` as the
   property type. The wait-then-start pattern ensures only one AX traversal runs
   at a time.

3. Update the comment on `detachedScrapeTask` to reflect that it is now accessed
   only from within `scrapeTask` (always on the MainActor cooperative thread).

Do not change the cancellation logic for `scrapeTask` itself — it is still
cancelled at the top of `presentPopup` so stale results are discarded. Do not
change `MenuScraper`, `PopupController`, or any other file. Do not bump the
version.
```
